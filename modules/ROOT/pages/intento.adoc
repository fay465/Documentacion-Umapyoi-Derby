== CONTEXTO

=== Problema del Negocio

[role="lead"]
*Problema Principal*: El proceso actual de devoluciones en la cadena de retail maneja:

* *Tiempo de procesamiento*: 3-5 días hábiles por devolución
* *Costo operacional*: $15-20 USD por devolución manual  
* *Tasa de error*: 12% en procesamiento manual
* *Insatisfacción cliente*: 35% de clientes reportan mala experiencia en devoluciones
* *Pérdida por fraudes*: 8% del total de devoluciones son fraudulentas no detectadas

*Necesidad crítica*: Automatización que reduzca costos operacionales y mejore la experiencia del cliente, manteniendo la capacidad de procesar 10M de transacciones diarias.

== HIPÓTESIS Y SUPUESTOS

=== Qué SÍ es el proyecto:

* *Sistema transaccional* de alta velocidad para gestión de devoluciones
* *Motor de reglas* para validación automática de políticas (5 dias para la devolucion)
* *Integrador* con sistemas de inventario, ventas y devoluciones

=== Qué NO es el proyecto:

* ❌ No es un sistema de ventas POS
* ❌ No es un ERP completo  
* ❌ No maneja logística física
* ❌ No reemplaza sistemas contables existentes

=== Supuestos técnicos validados:

. *Volumen*: 10'000.000 transacciones/día = ~2.777 transacciones/segundo (peak)
. *Disponibilidad*: Requerimiento 99.99% = ~52 minutos downtime/año máximo
. *Latencia*: <200ms para 95% de las transacciones

=== Flujos de Negocio Críticos

==== Flujo Principal de Devolución

*Inicio del Proceso*:
El flujo se activa cuando un cliente solicita una devolución a través de cualquiera de los canales habilitados: aplicación móvil, sitio web o directamente en tienda física. El sistema captura automáticamente la información básica de la transacción original.

*Validación Inicial y Crítica*:
La primera y única regla automática aplicada actualmente es la validación del período de 5 días desde la fecha de compra. El sistema calcula exactamente el tiempo transcurrido y determina si la solicitud cumple con este requisito fundamental.

*Escenarios de Validación:*

* Aprobado automático: Si la solicitud está dentro del período de 5 días

* Rechazo automático: Si ha transcurrido más de 5 días desde la compra

* Escalación manual: Casos especiales que requieren supervisión humana

*Proceso de Aprobación*:
Para las solicitudes que pasan la validación inicial, el sistema genera automáticamente un código QR único que sirve como comprobante digital. Este código contiene toda la información necesaria para procesar la devolución en tienda.

=== Requisitos funcionales, restricciones y dependencias


==== I. Carga de Trabajo Crítica (10M Tx/Día)

La gestión de 10 millones de transacciones diarias demanda una planificación rigurosa de la capacidad. El volumen transaccional en el sector minorista es inherentemente variable, con picos de demanda que superan con creces el promedio estadístico.

*   *Rendimiento Promedio:* Un volumen diario de 10,000,000 de transacciones se traduce en un rendimiento promedio constante de aproximadamente $115.7$ transacciones por segundo (TPS).
*   *Capacidad de Pico:* El dimensionamiento de la arquitectura debe basarse en la capacidad de manejar picos sostenidos. Se estima que, durante eventos críticos como lanzamientos de productos, ofertas estacionales o el Black Friday, la demanda puede escalar entre 4 y 10 veces el promedio. Por lo tanto, la arquitectura se diseñará para un pico sostenido de *500 TPS*, manteniendo la capacidad elástica para manejar ráfagas de hasta *1,000 TPS* sin degradación del servicio.
*   *Requerimientos de Datos:* En el contexto de bases de datos NoSQL como Azure Cosmos DB, estas 10 millones de operaciones diarias pueden requerir una línea base de aproximadamente $2,300$ Request Units por segundo (RU/s), asumiendo un costo conservador de 20 RU por transacción de escritura/lectura. La capacidad de pico debe ascender a $125,000$ RU/s, lo que justifica el uso de capacidades de escalamiento automático (Autoscale).

===0 II. Diseño Arquitectónico de Alta Disponibilidad y Marco de Cómputo

La plataforma se desplegará bajo un modelo de redundancia geográfica (HA/DR), utilizando Zonas de Disponibilidad (AZs) dentro de una región primaria y replicación activa-pasiva o activa-activa hacia una región secundaria. Este enfoque garantiza un SLA de tiempo de actividad superior.

==== II.A. Capa de Aplicación (Compute and Hosting)

La elección del servicio de cómputo para la lógica de negocio y la capa de presentación debe reflejar el compromiso con la máxima seguridad y rendimiento. Las dos opciones principales de Microsoft que cumplen con el criterio de alta inversión son Azure App Service Isolated v2 o Azure Kubernetes Service (AKS) Premium.

===== Azure App Service Isolated v2 (Recomendado para Arquitectura PaaS Pura)

La selección de App Service Isolated v2 se justifica por la necesidad de aislamiento de red y recursos dedicados. Este nivel proporciona entornos de servicio de aplicaciones dedicados y completamente aislados dentro de una red virtual de Azure (VNet), lo cual es crucial para requisitos regulatorios de seguridad y cumplimiento en el sector retail. [5]

Para gestionar picos de 500 a 1,000 TPS, es necesario un plan de servicio con instancias grandes que pueda escalar horizontalmente hasta el límite de 100 instancias ofrecido por el entorno Isolated. [8]

*   *SKU Base:* Se recomienda el plan de servicio I4v2. Esta es una de las SKUs más potentes disponibles en la capa Isolated v2, con 16 Cores y 64 GB de RAM por instancia. [9]
*   *Dimensionamiento Inicial y Costo Base:* Para la configuración base de alta disponibilidad en la región primaria, se dimensionan 4 instancias activas. Asumiendo que la aplicación utiliza Windows Server (a menudo el caso para migraciones.NET), se utiliza la tarifa más alta de pago por uso (PAYG) para este cálculo inicial. [9]
** Costo de I4v2 (Windows, PAYG): $\$ 3,282.08$ por mes. [9]
** *Costo de Cómputo Base:* $\$ 3,282.08 / mes \times 4$ instancias $\approx \$ 13,128.32$ por mes.

===== Azure Kubernetes Service (AKS) Premium (Opción Alternativa para Contenedores)

Si la estrategia de desarrollo se centra en microservicios y contenedores, AKS es la opción de misión crítica. Para cumplir con la alta disponibilidad y la capacidad de escala hasta 5,000 nodos, se requiere el nivel *Standard Tier*. [10] Sin embargo, dada la directriz de "no escatimar", se recomienda el *Premium Tier* por sus beneficios de soporte a largo plazo (LTS). [11]

*   *Costo del Plano de Control (Control Plane):* El Premium Tier se factura a $\$ 0.60$ por *cluster* por hora. [11]
** Costo Mensual del Control Plane (Premium): $\$ 0.60 / hora \times 730$ horas/mes $\approx \$ 438.00$ USD por *cluster*.
*   *Costo Real del Cómputo:* El costo dominante en AKS es el de los Nodos de Agente (VMs) que ejecutan los contenedores, que deberían ser dimensionados de forma similar a los recursos I4v2 (VMs D-series o similares). [12]

=== III. Especificación Detallada de Componentes PaaS de Misión Crítica

La arquitectura de alta inversión exige soluciones de datos que puedan crecer linealmente con la carga transaccional masiva.

==== III.A. Capa de Datos Transaccionales (OLTP Core)

El núcleo del negocio retail (pedidos, pagos, inventario) requiere una base de datos relacional que garantice la consistencia ACID. [3]

===== 1. Azure SQL Database Hyperscale

Azure SQL Database Hyperscale es la elección de misión crítica para esta escala. [6] Su principal ventaja es la separación de cómputo y almacenamiento, lo que permite un escalamiento de almacenamiento de hasta 100 TB sin un aumento proporcional de los recursos de cómputo, y sin afectar la latencia transaccional. [13, 14]

*   *Dimensionamiento de Cómputo:* Para un volumen de 500+ TPS sostenido, se requiere un alto compromiso de vCores, típicamente utilizando la serie Standard (Gen 5). Un punto de partida conservador, basado en el requerimiento de alto rendimiento, es 12 vCores.
** Costo de Cómputo (12 vCores): $\$ 3.3916$ por hora. [15]
** Costo Mensual (12 vCores, Solo Principal): $\$ 3.3916 / hora \times 730$ horas/mes $\approx \$ 2,475.80$ USD/mes.
*   *Alta Disponibilidad:* Para garantizar el SLA, se requiere la configuración de réplicas secundarias de alta disponibilidad (HA). [13] Esto efectivamente duplica el costo de cómputo.
** *Costo Mensual Total OLTP (2 Réplicas HA):* $\$ 2,475.80 \times 2 \approx \$ 4,951.60$ USD/mes (solo cómputo).

Una consideración financiera importante es que el almacenamiento de bases de datos en Hyperscale es escalable y el costo de *backup* está incluido inicialmente hasta el tamaño máximo de los datos. [13] Esto significa que el principal factor de costo recurrente es el componente de vCores, lo que dirige la estrategia de optimización financiera hacia los beneficios de licencia (como Azure Hybrid Benefit, discutido en la Sección VII).

===== 2. Azure Cosmos DB (NoSQL y Baja Latencia Global)

Azure Cosmos DB es fundamental para cargas de trabajo de retail que requieren latencia ultrabaja y escalamiento global para datos no relacionales (ej. perfiles de usuario, carritos de compra, catálogo de productos). La alta inversión en este servicio se justifica por su capacidad de ofrecer disponibilidad inigualable y replicación de escritura multi-región. [7]

*   *Dimensionamiento de Throughput:* Para soportar la línea base de 10 millones de operaciones de escritura diarias y absorber picos, se dimensiona un *throughput* de *10,000 RU/s* utilizando el modelo *Autoscale Provisioned*. [7] El uso de Autoscale es preferible al *Standard Provisioned* porque permite que la capacidad elástica absorba los picos de tráfico de retail de manera eficiente, optimizando el costo operativo al tiempo que garantiza el rendimiento. [7]
*   *Configuración de Alta Inversión:* Se recomienda la replicación en 4 regiones (Región principal + 3 regiones secundarias) para maximizar la redundancia geográfica (DR) y la latencia de lectura global.
*   *Costo Estimado (4 Regiones):* Basado en el ejemplo de replicación de 10K RU/s y 4 TB de almacenamiento distribuido (1 TB principal, 3 TB secundarios), el costo mensual de *throughput* y almacenamiento es de aproximadamente $\$ 3,404.80$ USD. [7, 16]

==== IV. Marco Integral de Seguridad y Observabilidad de Nivel Empresarial

La directriz de no escatimar en necesidades se aplica rigurosamente a la seguridad y la observabilidad, componentes críticos para proteger los datos de los clientes y mantener la continuidad operativa.

===== IV.A. Seguridad Perimetral y Mitigación de Ataques

====== 1. Azure DDoS Protection Standard

Dado que la empresa retail está expuesta al tráfico público de Internet, la protección contra ataques de Denegación de Servicio Distribuido (DDoS) es obligatoria.

*   *Costo Fijo:* El nivel *DDoS Protection Standard* impone un cargo fijo de $\$ 2,944$ por mes, que incluye protección para 100 recursos de IP pública. [23]

La adopción de este plan de alta inversión tiene un beneficio adicional de optimización de costos. Cuando DDoS Network Protection está habilitado en la VNet, el Web Application Firewall (WAF) desplegado en Application Gateway se factura a la tarifa de Application Gateway Standard v2, en lugar de la tarifa más alta de WAF. [23, 24] Este descuento parcial compensa marginalmente el alto costo base del plan DDoS.

====== 2. Azure Firewall Premium

Para garantizar la inspección de tráfico de salida y entrada más rigurosa, se requiere el nivel *Premium* de Azure Firewall, que ofrece capacidades avanzadas como inspección TLS y detección de intrusiones basada en inteligencia de amenazas. [25, 26]

*   *Costo de Despliegue:* $\$ 1.75$ por hora por despliegue. [26]
*   *Dimensionamiento:* Se requieren 2 despliegues en modo de Alta Disponibilidad (HA) para la región primaria.
*   *Costo Anual de Despliegue:* $\$ 1.75 / hora \times 730$ horas/mes $\times 12$ meses $\times 2$ despliegues $\approx \$ 30,660.00$ anual, o $\sim \$ 2,555.00$ mensual. Los costos de procesamiento de datos se añaden a esta tarifa, a $\$ 0.016$ por GB procesado. [26]

==== IV.B. Monitoreo, Log Analytics y SIEM (Microsoft Sentinel)

La capacidad para monitorear, detectar y responder a amenazas en tiempo real es vital. Esto se logra mediante la plataforma SIEM nativa de Microsoft, Sentinel. [27]

*   *Volumen de Logs:* Un sistema que procesa 10 millones de transacciones diarias, genera un tráfico masivo de logs de aplicación, base de datos y red. Un volumen conservador y realista para una operación de esta escala, que no escatima en la ingesta de logs de seguridad y operacionales, es de *500 GB por día*. [28, 29]
*   *Nivel de Compromiso (Commitment Tier):* Para gestionar este gran volumen de datos de manera predecible y costo-efectiva, se debe seleccionar un Nivel de Compromiso fijo en lugar del modelo *Pay-As-You-Go*. [27]
*   *Optimización del Costo de Logs:* Si bien la ingesta de 500 GB/día representa un gasto significativo, el modelo de Nivel de Compromiso proporciona un ahorro sustancial, hasta el 60% en comparación con la tarifa PAYG. [28]
** Costo Diario (500 GB/día Commitment Tier): $\$ 492$ USD/día. [28]
** *Costo Mensual Proyectado de Sentinel:* $\$ 492 / día \times 30$ días $\approx \$ 14,760$ USD/mes.

Esta estrategia asegura que la seguridad y el monitoreo no se vean limitados por preocupaciones presupuestarias a corto plazo, ya que la ingesta masiva se convierte en un costo operativo fijo y altamente descontado.

=== Criterios de aceptación

Desde la perspectiva del cliente, el sistema se considera exitoso cuando permite completar una solicitud de devolución en menos de tres minutos, con confirmación inmediata y visibilidad completa del estado del trámite en todo momento. La experiencia debe ser consistentemente excelente a través de todos los canales digitales.

Para el personal de tienda, la solución debe demostrar su valor mediante la reducción significativa del tiempo de procesamiento por caso, el acceso inmediato a información relevante y la capacidad de resolver excepciones comunes sin escalamiento. La integración con los flujos de trabajo existentes debe ser fluida e intuitiva.

==== Eleccion del lenguaje Elixir para la programacion de la API

* *Status:* Acepted
* *Fecha de creacion:*  24/10/2025
* *Fecha ultima modificacion:*  24/10/2025
* *Participantes:*
** @PolloDiego159
* *Decisores:*
** @clsource
* *Contexto:*
** El sistema requiere procesar 10 millones de transacciones diarias con latencias consistentes menores a 200ms. La naturaleza concurrente de las solicitudes de devolución y la necesidad de alta disponibilidad exigen un lenguaje que maneje eficientemente la concurrencia masiva. El equipo actual tiene experiencia mixta en diferentes lenguajes y frameworks.
* *Lista de opciones y alternativas*
** *Elixir:* Es un lenguaje de programación funcional, concurrente, de propósito general que se ejecuta sobre la máquina virtual de Erlang. Elixir está escrito sobre Erlang y comparte las mismas abstracciones para desarrollar aplicaciones distribuidas y tolerantes de fallos.
** *Kotlin:* Es un lenguaje de programación orientado a objetos multiplataforma, estáticamente tipado, de alto nivel y propósito general con inferencia de tipos.
** *Switf:* Es un lenguaje de programación moderno y de código abierto creado por Apple para desarrollar aplicaciones en sus plataformas, como iOS, macOS, watchOS y tvOS. Se caracteriza por ser rápido, seguro, fácil de usar y por tener una sintaxis limpia y expresiva que mejora la experiencia del desarrollador en comparación con su antecesor, Objective-C. 
* *Decisión* 

*Se eligio Elixir*

** El principal motivo fue el facil uso de elixir y el uso con el framework Phoenix y Liveview
** El proceso de aprendizaje fue mas facil de entender desde el desarrollo con Elixir
* *Observaciones y Notas*
** Fue bastante complicado entender al inicio el proyecto debido a que el equipo (yo) no entendia muy bien el lenguaje ademas de los nuevos conceptos que se aplicaron para el desarrollo de la API
** Se debe probar en un entorno mas demandante para el caso especifico de que en realidad deba soportar 10 millones de transacciones diarias
* *Preguntas abiertas*
** *¿Volveria a desarrollar una API con elixir?:* Despues de todo el trabajo es probable que no vuelva a desarrollar una API :c
** *¿Todas las funciones se implementaron?:* No, particularmente se queria agregar un apartado de comentario al hacer la devolucion lo cual no fue posible porque cada que se intento agregar dicha funcion todo dejaba de funcionar, por lo que se opto por omitirlo.

==== Eleccion del gestor de base de datos

* *Status:* Proposed
* *Fecha de creacion:*  24/10/2025
* *Fecha ultima modificacion:*  24/10/2025
* *Participantes:*
** @PolloDiego159
* *Decisores:*
** @clsource
* *Contexto:*
** El sistema de devoluciones maneja datos transaccionales críticos que requieren consistencia ACID fuerte. Las operaciones incluyen procesamiento financiero (reembolsos), gestión de inventario y trazabilidad completa de cada devolución. Los requerimientos regulatorios exigen integridad absoluta de los datos financieros.
* *Lista de opciones y alternativas*
** *PostgreSQL:* base de datos relacional open-source con fuerte compliance ACID y amplias capacidades.
** *Microsoft SQL Server:* Es un sistema de gestión de base de datos relacional, desarrollado por la empresa Microsoft. El lenguaje de desarrollo utilizado es Transact-SQL, una implementación del estándar ANSI del lenguaje SQL, utilizado para manipular y recuperar datos, crear tablas y definir relaciones entre ellas.
** *Oracle Database:* Es un sistema de gestión de base de datos de tipo objeto-relacional, desarrollado por Oracle Corporation, la empresa estadounidense de hardware y software. Este tipo de sistema mejora la gestión de grandes bases de datos y programación y también aumenta el nivel de seguridad.
* *Decisión* 

    *Se eligio Microsoft SQL Server*

** El principal motivo fue que por temas de competencia de marcas era mejor contratar los servicios con Azure y, de esa forma y unificando todo el ecosistema, tambien contratar la nube de Microsoft
** Cuenta con herramientas de management excelentes demas de tener buen performance y estabilidad, ademas de un sistema de soporte enterprise
* *Observaciones y Notas*
** Aunque en el proyecto actual no se use este sistema de base de datos se recomienda debido a la exigencias que requiere el negocio
** En el caso de que se quiera utilizar la nube de azure para implementar todo el proceso de la API resulta muy comodo manejar el mismo entorno, por lo que se espera que todo las demas depencias tambien actuen en un entorno de Microsoft
* *Preguntas abiertas*
** *¿Puede usarse el mismo modelo que se esta usando en la prueba?* Al ser los dos sistemas SQL no deberia haber mayores diferencias al usar un entorno SQLite o la nube de Microsoft
